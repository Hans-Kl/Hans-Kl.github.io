Events or Commands[译文]

发表于 2021-05-17 Valine： 6.6k 6 分钟

前言：本篇探讨命令与事件各自优缺点和如何选择，原文地址：https://codesimple.blog/2018/04/16/events-or-commands/

事件还是命令

Most of the programmers quickly agree that naming is one of the hardest things they have to do as part of their Job.大多数程序员迅速同意命名是他们作为工作的最艰难的事情之一。

As Phil Karlton puts it “_There are only two hard things in Computer Science: cache invalidation and naming things._“正如菲尔卡尔顿所说的那样，“计算机科学中只有两个艰难的东西：缓存失效和命名的东西。”

Naming is hard because it is important to get it right. It is one of the key UX for readers/consumers of your software components. There is plenty of material (books/blogs) written on this topic. I have also shared my thoughts in another post.命名很难，因为重要的是要正确。它是软件组件的读者/消费者的关键UX之一。有大量的材料（书籍/博客）写在这个主题上。我还在另一个帖子中分享了我的想法。

Besides other challenges, developers of Event-Driven systems are suffering from this pain at another level. While building an event-driven system, generally we find it difficult to name an event. Often in this situation, if the name of the event you are raising does not resonate the business event then you should be alarmed and reassess the event.除其他挑战外，活动驱动系统的开发人员正在患有这种痛苦的痛苦。在构建事件驱动的系统时，通常我们发现难以命名事件。经常在这种情况下，如果您提出的事件的名称不会产生业务事件，那么您应该被警告并重新评估事件。

Martin Fowler warns about the trap of Passive Aggressive Commands:Martin Fowler警告了被动侵略性命令的陷阱：

“This happens when the source system expects the recipient to carry out an action, and ought to use a command message to show that intention, but styles the message as an event instead.”“当源系统期望收件人执行一个动作时，会发生这种情况，并应该使用命令消息来显示该意图，但是改变为事件的消息。”

Before digging deeper, lets define commands and events.在挖掘更深时，允许定义命令和事件。

Today I am taking a look at several complementary concepts that fall under the Event Driven Architecture (EDA) umbrella. I have been working as an Architect in a team that is starting their EDA journey. I compiled this terminology explanation for them to help unpack the many connected concepts.今天我正在看看几种互补概念，该概念落在了活动驱动的建筑（EDA）伞下。我一直在一支开始他们的eda旅程的团队中担任建筑师。我编译了这个术语解释，以帮助解压缩许多连接的概念。

Some of these concepts govern a high-level macro architecture (how our system hangs together) and others are closer to code, micro architecture (what our system does). We can consider the macro architecture the Big Boxes, and how they work together, whereas the micro architecture governs what is happening inside the box.其中一些概念管理高级宏体系结构（我们的系统如何挂起），其他概念更接近代码，微架构（我们的系统）。我们可以考虑宏观体系结构大盒子，以及它们如何一起工作，而微型架构管理盒子里面发生的事情。

What do you mean by “Event-Driven”?[译文]

发表于 2021-05-14 Valine： 12k 11 分钟

前言：这个系列是将自己看过的一些国外大佬的文章，翻译整理一下，避免以后看的时候还得再看一遍英文。
本篇作为第一篇是 Martin Fowler 在博客中探讨有关EDA(事件驱动架构)的文章，原文地址：https://martinfowler.com/articles/201701-event-driven.html

Towards the end of last year I attended a workshop with my colleagues in ThoughtWorks to discuss the nature of “event-driven” applications. Over the last few years we’ve been building lots of systems that make a lot of use of events, and they’ve been often praised, and often damned. Our North American office organized a summit, and ThoughtWorks senior developers from all over the world showed up to share ideas.

在去年年底，我与ThoughtWorks的同事一起参加了一个研讨会 思考讨论“事件驱动”应用的本质。在最近几年里，我们一直在建立大量的系统，这些系统大量使用了事件，并且他们毁誉参半。我们的北美办公室组织了一个峰会，来自世界各地的ThoughtWorks的高级开发与会并分享想法。

The biggest outcome of the summit was recognizing that when people talk about “events”, they actually mean some quite different things. So we spent a lot of time trying to tease out what some useful patterns might be. This note is a brief summary of the main ones we identified.

峰会的最大结果是：我们认识到当人们谈论 “事件”，他们实际上意味着一些完全不同的东西。所以我们花了很多时间，试图整理出有用的模式可能是什么。本说明是对我们确定的主要部分的一个简短的摘要。

Event Notification事件通知

This happens when a system sends event messages to notify other systems of a change in its domain. A key element of event notification is that the source system doesn’t really care much about the response. Often it doesn’t expect any answer at all, or if there is a response that the source does care about, it’s indirect. There would be a marked separation between the logic flow that sends the event and any logic flow that responds to some reaction to that event.

“事件通知”发生在：一个系统发出事件消息以通知其他系统他自己的领域内发生了某种变化。事件通知的一个关键元素是 源系统对响应并不关心。通常源系统根本不期望任何响应，或者即使源系统确实关心响应，但响应是间接的。在发送事件的逻辑流程和任何对该事件作出反应的响应之间，应该是标记分离的。

Event notification is nice because it implies a low level of coupling, and is pretty simple to set up. It can become problematic, however, if there really is a logical flow that runs over various event notifications. The problem is that it can be hard to see such a flow as it’s not explicit in any program text. Often the only way to figure out this flow is from monitoring a live system. This can make it hard to debug and modify such a flow. The danger is that it’s very easy to make nicely decoupled systems with event notification, without realizing that you’re losing sight of that larger-scale flow, and thus set yourself up for trouble in future years. The pattern is still very useful, but you have to be careful of the trap.

谈谈 Java 的异常机制

发表于 2020-06-07 分类于 基础 Valine： 2.4k 2 分钟

お前も舞うか? ——宇智波斑 《火影忍者》

为什么会有异常

远古时期的编程语言中是没有”异常”这个概念存在的，大家对于非正常的情况会约定一个错误标识，由下层方法返回给上层调用方法，类似ErrorCode = 400这样，但是这样做有几个明显的缺点：

  1. 调用方需要额外编写大量检查错误标识的代码，并且这些代码与调用方本身的业务代码紧耦合在一起
  2. 调用方不一定有能力处理这些异常，如果没有能力处理的话，该怎么处理错误标识就变成了一个棘手的问题，是将错误标识继续向上传递还是忽略错误码继续自己的流程呢？
  3. 错误标识将处在同一个调用链上的各层方法紧耦合在了一起，比如底层方法抛出的错误标识如果有修改，上层方法全部都要兼容

总的来说就是耦合耦合太过紧密，针对这几个问题，产生了现在语言的“异常”体系。

现代的异常允许底层抛出的异常，跨越数个调用方法，一直到最顶层的方法，这样耦合性就降低了，没有能力异常的调用方不需要非得对异常进行处理，可以简单的向上抛出。
