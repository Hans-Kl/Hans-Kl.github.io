Java NIO使用的是IO多路复用模型，IO多路复用模型与传统IO模型的区别在于：一个线程可以检查多个fd，而传统IO模型中一个线程只能检查一个文件描述符fd。

IO多路复用模型用三种常见的实现方式：

Redis系列——过期处理

发表于 2021-08-02 分类于 redis Valine： 2.6k 2 分钟

前言

最近又研究起了Redis，被Redis作者Antirez折服了，redis源码的优雅和搞笑不必多说，文档和注释的详细更让我吃惊，作为一个开源项目，有这样的耐心是令人敬佩的。

本篇主要聚焦于Redis对于过期键的处理。

常见的过期处理策略

在我们日常的业务开发中，也会经常有数据到期之后自动失效的需求，一般由三种实现方案：

  1. 定时删除，在设置数据过期时间的时候，设置一个定时器，定时器会在数据过期之后立即将数据删除掉。

  2. 惰性删除，不主动扫描数据是否过期，而是在数据再次被使用时，判断数据是否已经过期了，如果过期就将数据删除掉，没有过期则继续执行处理。

  3. 定期删除，使用一个后台任务，定期扫描数据，发现有过期的数据就删除掉。

分析一下三种实现方案的优劣势：

  1. 定时删除是内存友好，但CPU不友好的：无效的内存会在数据过期后立即被释放，腾出内存空间，但如果过期的key数量很多，会导致cpu飙高，影响主线程读写数据，所以Redis没有采用这种方案。
  2. 惰性删除是CPU友好，但内存友好的：不需要额外的清理过期数据线程，但如果客户端一直没有查询过那些过期的数据，则会导致redis-server内存泄露，除非手动执行flushdb。Redis为了不影响性能，使用了这种方式。
  3. 定期删除是中和的，更像是一种妥协。用一个线程扫描设置过期时间的数据字典，执行清理任务。Redis对于执行定期删除的方案进行了细致的优化。

JVM系列——从新垃圾到老垃圾

发表于 2021-07-02 分类于 jvm Valine： 729 1 分钟

前言

我们知道，对象会在年轻代的Eden区中分配内存空间，等对象不再使用后，就会变成垃圾。有的垃圾会被顺利回收，而有的垃圾则会进入老年代Old，变成老垃圾，本篇文章笔者就探讨一下，什么样的狡猾的垃圾，才会从年轻代升级到老年代中去👴🏻。

1. 达到年龄阈值

当存活的对象，连续躲过15次young gc后，它就变成了15岁的对象，这时称这些对象达到了年龄阈值，这种老对象会进入老年代。

默认是15岁，这个年龄可以用-XX:MaxTenuringThreshold参数更改。

2. 动态年龄判断

https://segmentfault.com/a/1190000039805691

3. 大对象直接进入老年代

我们可以手动设置一个参数：-XX:PretenureSizeThreshold= <byte size>，这个参数代表对象大小，单位是字节。设置了这个参数之后，超过这个字节数的对象会直接在老年代分配内存，而不会在Eden区分配。默认值是0，代表所有对象都现在Eden区分配内存。

客户端负载均衡

Ribbon和前面研究的Eureka组件不太一样，他不需要独立部署，而是依附于客户端应用，随着客户端一起部署。在调用方客户端向外发起请求的时候，代理发起请求的操作，为调用方选择一个合理的提供方实例。

负载均衡与服务实例

我们之前接触到的负载均衡，一般都是在说服务端负载均衡，服务端负载均衡有硬件和软件之分，不过原理是类似的，有感兴趣的同学可以自行了解一下。凡是具备负载均衡功能的组件，都需要维护服务实例注册表，要能感知到所有服务实例的状态，才能通过实例的健康度和自定义配置来进行请求的负载均衡。通常通过keep-alive等心跳机制保持与服务实例的通信

“顺手牵羊”注册表

注册表？听着那么熟悉呢。嗷，Eureka已经为我们维护了现成的注册表和服务实例状态，更贴心的是，还维护了客户端的缓存注册表。作为负载均衡的组件，我们只要借用Eureka的这套实现就可以啦！

并且既然已经有客户端的缓存注册表了，那我们不如将计就计，不在服务端做负载均衡，而是改成：在客户端发起调用的时候，对服务提供方的实例进行负载均衡，这样就能利用上客户端本地的缓存注册表了
